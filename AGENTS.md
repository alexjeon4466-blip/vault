# AGENTS.md — 북클럽 × 희곡 창작 LLM Wiki v2 운영 규칙

이 vault는 사용자의 비공개 창작-강독 순환 장치다. 매달 세 권씩 쌓이는 강독 자료의 휘발성을 막고, 전혀 연결 없어 보이는 책들 사이의 질문·정서·주제를 발견하며, 그 에너지가 단막극의 주제와 퇴고 감각으로 돌아오게 한다.

## 0. 최우선 원칙

- 기본 언어는 한국어다.
- 사용자가 실제 강독에서 말하는 듯한 구어체를 우선한다.
- 이 vault는 공개용 저장소가 아니라 비공개 작업실이다.
- `bookclub/`과 `script/`는 원본 자료로 간주하고 수정하지 않는다.
- 기존 v1 위키는 `_archive/wiki_v1/`에 보존되어 있다. 참고는 가능하지만, v2 문서는 원본에서 새로 작성한 뒤 v1과 비교한다.
- 여러 파일을 만들거나 수정하기 전에는 대상 파일 목록을 분명히 한다.

## 1. 이 위키의 목적

이 위키는 단순한 독서 기록이 아니다.

1. **강독의 기억 장치** — 사전리서치, 작가론, 시대배경, 인용문 분석, 현재 시사점을 한 권의 이해로 붙잡는다.
2. **책 사이 연결 지도** — 주제별·정서별·질문별 연결을 만든다.
3. **희곡 창작 작업실** — 반복 질문이 단막극의 사건, 장면, 대사, 퇴고 감각으로 돌아오게 한다.
4. **콘텐츠화 기반** — 강의형 유튜브/팟캐스트/발제문으로 바꿀 수 있는 구조를 남긴다.

## 2. 쓰기 허용 범위

명시적 허락 없이 쓸 수 있는 곳:

- `wiki/` 아래 새 v2 문서
- `output/` 아래 초안/리포트
- `_templates/` 아래 템플릿
- `SCHEMA.md`, `index.md`, `log.md`, `LLM_WIKI_PLAYBOOK.md`, `CLAUDE.md`, `AGENTS.md`

명시적 허락 없이는 금지:

- `bookclub/` 원본 수정/삭제/이동
- `script/` 원고 수정/삭제/이동
- `_archive/wiki_v1/` 수정/삭제
- 기존 원본 파일명 변경
- 대량 폴더 이동

## 3. 문체 규칙

- 너무 논문식으로 쓰지 않는다.
- 책을 읽은 사람에게 다시 설명하는 강의형 문장으로 쓴다.
- `내 해석`은 사용자가 실제로 붙잡았을 법한 문제의식으로 쓴다.
- AI가 적극적으로 연결/가설을 제안할 수 있지만 `AI 가설`, `가능한 연결`처럼 분리한다.
- 확신이 낮은 내용은 `추정`, `확인 필요`, `가설`로 표시한다.

## 4. 북클럽 작업 규칙

책카드는 다음을 우선한다.

- 작가는 왜 이 책을 썼을까
- 이 책의 핵심 질문은 무엇인가
- 강독 흐름은 어떻게 잡히는가
- 내 해석은 무엇인가
- 오래 남겨야 할 장면/문장은 무엇인가
- 다른 책과 어떤 질문/정서로 연결되는가

줄거리 요약은 보조다. 이미 북클럽이 끝난 책이므로, 핵심 질문과 연결을 우선한다.

## 5. 희곡 작업 규칙

희곡카드는 작품 소개가 아니라 퇴고 감각의 저장소다.

항상 확인한다.

- 주제는 있는데 사건이 약하지 않은가?
- 관객이 이해할지 불안해서 설명이 과해지지 않았는가?
- 설명을 덜어내면서 작품의 미학까지 해치고 있지는 않은가?
- 이 작품은 무슨 질문을 가진 이야기였는가?
- 다음 퇴고는 의도와 주제를 더 살리는가?

## 6. 질문형 노트 우선

추상 명사보다 질문형 노트를 우선한다.

나쁜 예:
- 제도
- 관계
- 언어

좋은 예:
- [[제도는 인간을 어떻게 지우는가]]
- [[언어는 왜 관계를 구하지 못하는가]]
- [[자각은 왜 사건이 되지 못하는가]]

질문형 노트는 책과 책, 책과 희곡을 연결하는 중심 노드다.

## 7. Index / Log 규칙

- 새 wiki 문서를 만들면 `index.md`에 추가한다.
- 모든 의미 있는 작업은 `log.md`에 append-only로 기록한다.
- `log.md` 기존 항목은 수정하지 않는다.

## 8. 파일 우선 / convention 우선 원칙

- 이 vault의 장기 산출물은 앱이나 캐시가 아니라 `.md` 파일이다. 임시 변환물, transcript chunk, 분석 cache는 언제든 재생성 가능한 것으로 취급한다.
- 새 `wiki/` 문서는 `SCHEMA.md`의 frontmatter convention을 따른다. 특히 `type`, `status`, `sources`, `connected_questions`를 우선 채우고, 관계가 명확할 때만 `related_to` / `belongs_to` / `has`를 추가한다.
- `_`로 시작하는 frontmatter 필드는 워크플로·검증·생성 출처 같은 내부 메타데이터다. 해석 본문처럼 읽히지 않게 최소한으로 쓴다.
- 로컬 API 키, Obsidian REST URL, 임시 media/cache 경로처럼 특정 PC에만 속하는 값은 vault에 쓰지 않는다.
- 자동화나 MCP 작업은 vault root 밖 파일을 만들거나 덮어쓰지 않는다. 새 노트 생성은 기존 파일 overwrite 없이 수행한다.

## 9. v2 첫 구축 원칙

첫 기준 문서는 다음이다.

- `시지프신화`
- `부활`
- `관계란`
- 질문형 노트: 부조리, 자각, 제도, 언어/관계 계열

이후 매달 새 책이 들어오면 책카드 → 질문형 노트 → 연결 지도 순서로 확장한다.
