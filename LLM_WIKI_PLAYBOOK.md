# LLM_WIKI_PLAYBOOK.md — v2 작업 매뉴얼

## 1. 첫 방향

이 위키는 강독의 휘발성을 막고, 책 사이 연결을 발견하며, 그 연결이 희곡 창작과 퇴고로 돌아오게 하는 비공개 LLM Wiki다.

## 2. 새 책카드 작성 절차

1. 대상 책 폴더를 확인한다.
2. `파트2`, `통합`, `최종`, 발표용 스크립트에 가까운 파일을 우선 읽는다.
3. 필요하면 사전리서치/작가소개/시대배경 파일을 보조로 읽는다.
4. 기존 v1 문서는 먼저 보지 않는다.
5. 원본 기반으로 v2 책카드를 작성한다.
6. `_archive/wiki_v1/`의 같은 책 문서를 읽고 빠진 통찰만 비교 반영한다.
7. 연결 가능한 질문형 노트를 만들거나 갱신한다.
8. `index.md`와 `log.md`를 갱신한다.

## 3. 책카드 v2 프롬프트

```text
다음 책 폴더를 원본으로 읽고 v2 책카드를 작성해 주세요.

목표:
- 줄거리 요약보다 핵심 질문, 내 해석, 작가가 왜 이 책을 썼는지, 오래 남는 장면, 다른 책과 연결되는 질문을 우선합니다.
- 문체는 한국어 구어체에 가까운 강독 톤입니다.
- `내 해석`과 `AI 가설`을 구분합니다.
- 모든 근거 파일을 남깁니다.
- 책마다 희곡 연결을 억지로 만들지 않습니다. 다만 질문형 노트로 자연스럽게 연결합니다.
```

## 4. 희곡카드 v2 프롬프트

```text
다음 희곡 원고/버전들을 읽고 v2 희곡카드를 작성해 주세요.

목표:
- 작품 소개보다 이 작품의 핵심 질문과 미학을 붙잡습니다.
- 주제는 있는데 사건이 약한지 확인합니다.
- 관객 이해를 불안해해서 설명이 과해진 부분과, 덜어내도 되는 부분을 나눕니다.
- 퇴고가 작품의 고유한 미학을 해치지 않는지 점검합니다.
- 다음 퇴고에서 실험할 선택지를 제안합니다.
```

## 5. 질문형 노트 작성 절차

질문형 노트는 책과 책, 책과 희곡을 연결하는 중심 노드다.

작성 기준:
- 최소 두 개 이상의 책/희곡이 같은 질문에 닿을 때 생성한다.
- 단, v2 핵심 질문으로 확실하면 첫 문서부터 생성해도 된다.
- 추상 명사보다 질문형 제목을 쓴다.
- 연결된 책들이 같은 말을 하는지, 서로 반박하는지, 다른 결말을 내는지 비교한다.

## 6. v1 비교 절차

v2 문서를 원본에서 작성한 뒤에만 v1을 비교한다.

확인할 것:
- v1에만 있는 좋은 통찰
- v1의 AI식 일반화 중 v2에서 버릴 것
- v1에서 억지로 만든 연결
- v2 질문형 노트로 승격할 만한 연결

## 7. Lint 체크리스트

- 책카드가 질문형 노트에 연결되어 있는가?
- 질문형 노트가 너무 추상적이지 않은가?
- `내 해석`과 `AI 가설`이 구분되어 있는가?
- 책 사이 연결이 인사이트풀한가, 단어만 같은 억지 연결인가?
- 희곡카드는 사건의 힘과 미학을 함께 보고 있는가?
- 원본 근거가 남아 있는가?

## 8. Frontmatter / QA convention

Tolaria에서 흡수한 원칙을 이 vault에 맞게 축약 적용한다.

- **파일 우선**: 장기 지식은 `wiki/`의 markdown 파일에 남긴다. cache, 임시 변환물, transcript chunk는 재생성 가능한 보조물이다.
- **Convention over configuration**: 새 wiki 문서는 `SCHEMA.md`의 `type`, `status`, `sources`, `connected_questions`를 기본으로 채운다. 관계가 명확한 경우 `related_to`, `belongs_to`, `has`를 Obsidian wikilink 배열로 쓴다.
- **시스템 필드**: `_workflow_version`, `_source_kind`, `_generated_by`, `_verification`처럼 `_`로 시작하는 필드는 내부 메타데이터다. 과도하게 늘리지 않는다.
- **로컬 설정 분리**: API key, Local REST URL, 임시 media/cache 경로, 도구 실행 상태는 vault 문서가 아니라 Hermes 설정/환경변수/로컬 작업 폴더에 둔다.
- **QA 우선순위**: 한글 IME/다국어 검색, wikilink 해석, frontmatter roundtrip, Windows path, 원본 markdown 보존을 자동화 변경의 주요 회귀 포인트로 본다.
- **쓰기 안전**: 자동화는 `bookclub/`, `script/`, `_archive/wiki_v1/`를 수정하지 않고, 기존 wiki 파일을 덮어쓰기 전에 read→patch 방식으로 변경 범위를 확인한다.
